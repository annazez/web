Mental Health App

MENTAL-HEALTH-APP-01: Data Privacy and Clinical Transparency

Context

The application handles Category I sensitive health data under GDPR. Users sharing intimate details about their mental state must have absolute confidence in how that data is used, stored, and - critically - never sold. Trust is not a UX feature; it is the product.

Decision

A privacy-by-design architecture with full local data processing where possible. Explicit, granular consent flows for any data that leaves the device. A clear anonymisation pipeline for any research data contribution, with opt-in mechanics and a plain-language data usage explanation. No dark patterns. No default data sharing. The anonymisation layer separates PII from clinical data at the point of ingestion, before any aggregation occurs.

Consequences

Reduced flexibility for real-time cloud-based personalisation features in early versions. Higher initial engineering investment in the local-first data layer. Significant long-term advantage in user trust, regulatory compliance, and suitability for integration into certified clinical systems.

MENTAL-HEALTH-APP-02: Clinically-Grounded Personalisation Engine

Context

Generic wellness apps offer generic advice. This application needs to deliver contextually appropriate content to users in genuinely varied psychological states - from mild stress management to active mood episodes. A one-size response is actively harmful in this domain.

Decision

The personalisation engine is seeded by three validated clinical instruments: WHO-5 (general wellbeing), GAD-7 (anxiety severity), and PHQ-9 (depression screening). These questionnaires are administered periodically and at contextually appropriate moments (not as intrusive popups). The resulting scores form a dynamic user state model that governs content recommendations, exercise difficulty, diary prompt tone, and notification frequency. AI-driven recommendation adjustments are made transparently, with users able to inspect why a particular recommendation was surfaced. The engine does not diagnose; it contextualises. Clinical interpretation remains the domain of qualified professionals within the integration layer.

Consequences

More complex content structure and recommendation logic than a simple rule-based system. Requires an ongoing clinical advisory relationship to validate scoring thresholds and response protocols. The tradeoff is a system that provides genuinely personalised support rather than performing personalisation as a marketing feature.

Telperion

TELPERION-01: Static Site Generation via Astro

Context

The NGO required exceptional performance, an extremely low carbon footprint, and a hosting cost that trends toward zero. A dynamic server-rendered site would introduce operational overhead the client could not sustainably absorb.

Decision

Astro as the SSG framework with zero client-side JavaScript by default.

Consequences

Sub-second load times on constrained connections. No backend attack surface - the entire output is a folder of pre-rendered HTML, CSS, and optimised assets. Hosting on a static CDN eliminates server management entirely. The architectural choice directly maps to the NGO's own values around sustainability.

TELPERION-02: Native i18n Routing and CI-Enforced WCAG Compliance

Context

Telperion operates across a bilingual (Czech and English) audience, including users with visual impairments who depend on screen readers. Accessibility could not be an afterthought bolted on at the end; it had to be load-bearing.

Decision

Astro's built-in i18n routing for language separation. WCAG 2.1 AA compliance is validated automatically on every build via Lighthouse and axe-core. The pipeline blocks any code-level PR that introduces contrast failures, broken ARIA labelling, or heading structure regressions - this applies to developer changes, not to the day-to-day content edits that non-technical staff make through Markdown.

Consequences

The language-aware URL structure (`/en/`, `/cs/`) is resolved at the framework level with no client-side routing overhead. Structural accessibility is protected at the point where it's most likely to break - architectural changes - without adding friction to routine content updates.
